Ingredient quantity — UX variants

Today the add-ingredient bar says "Qty/person" tersely and gives no live feedback. Joe typed 12 eggs thinking "12 total for the group" — 4 people attending, so grocery list showed 48 eggs. Three fixes below. All three keep per-person as the stored data model (so totals re-scale when headcount changes).

What's there today (the bug)
Result: 48 eggs on grocery list (silently multiplied × 4 attending)
Variant A — live preview
Smallest change. Keep per-person input, add an explicit "/ person" badge inside the qty input, and a preview chip that shows what the grocery list will actually get.
/ person
📊 = 8 eggs on grocery list · 4 attending
Pros: tiny visual change, no behavior change, preview makes the per-person math obvious without the user having to read a label.
Cons: user still has to do the mental division ("if I need 8, I enter 2").
Variant B — segmented toggle
Let the user pick how they think. Toggle between Per person and Total for group. We convert to per-person at save time so the data model doesn't change.
📊 = 8 eggs on grocery list · 4 attending
↓ saves as 2 / person (8 ÷ 4 attending)
Pros: matches however the user is thinking. Two-line recipes say "1 stick of butter" (total); bulk meals say "2 eggs each" (per-person). Toggle lets them type either.
Cons: extra control, small UI complexity bump. If headcount is 0 ("unplaced" meal), total mode can't convert — needs guardrail.
Variant C — total-first (simplest mental model)
Drop per-person entry entirely. User types the total needed. We divide by headcount at save time. If headcount changes later, we re-scale — with a warning so nobody is surprised.
total
Goes on grocery list as-is. If more people join later, we'll scale up proportionally and show a nudge.
Pros: matches how most people shop ("I need 8 eggs for this breakfast"). Zero math.
Cons: "scale on headcount change" is invisible magic. If Joe invites two more people after entering 8 eggs, grocery list becomes 12 — which may or may not match his intent. Needs an explicit "fixed total" vs "scales with attendance" switch to be honest about behavior.
My recommendation

Variant B. The live preview is the key fix from A, the toggle gives people the right mental model without changing the stored data, and it costs one tiny segmented control on each form (add bar + edit modal). Variant A is fine if you want the absolute minimum change.

Either way, the same picker component gets reused in the edit-ingredient modal (issue #3), so the "am I entering per-person or total?" question is answered in both places consistently.
