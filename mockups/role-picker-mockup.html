The RSVP-energy wizard — fires per trip, not per account

🎯 The Idea in One Line

Every trip, every person picks one of four energy levels. The pick drives how much onboarding they see, which tab they land on, and how loud the app gets. Planners get the full experience; reluctant downloaders get a Venmo-grade app that respects their time.

Design notes • Invite banner at the top reminds them why they're here — keeps the emotional context of "Dan asked me to download this" visible. • Energy-level labels, not role titles — "All In" doesn't out-rank "Vibes Only." They're just different stances. Removes the pecking-order feel entirely. • Self-deprecating copy validates every pick including the reluctant ones. The roast lands and then the app delivers on it. • Perk hint appears on selection so the user sees the payoff ("skip packing prefs, straight to the trip") before they commit. • "Skip →" in the top-right is a safety valve — never block a user from entering the trip. Default them to Helping Out and let them re-pick later in Group. • Follows the existing sticky-bottom-nav pattern from tab-layout-standard.md.

Where the Role Picker Fires — 3 Entry Flows

Flow 1 — Invited user (the hardest case)

1
Tap invite link → /invite/[code]
2
Lightweight signup (name, email, password) ~30 seconds
3
Role Picker fires → /trip/[id]/role NEW
4a
If All In / Helping Out: run full 8-step onboarding profile, style, packing prefs, friends
4b
If Just Here / Vibes Only: skip onboarding entirely — capture name only, drop into trip skips 6 steps
5
Land on /trip/[id] with sub-nav + default tab set by pick

Flow 2 — Existing user creates a trip

1
/trip/new → enter destination, dates, type
2
Save trip → Role Picker fires, pre-selected All In 🔥 "Looks like you're running this one. Sound right?"
3
One-tap confirm or re-pick
4
Land on /trip/[id] with All In UI density

Flow 3 — Existing user accepts invite to new trip

1
Tap invite (already signed in)
2
Accept modal on /trip/[id]/group
3
Role Picker fires — pre-filled with smart default based on last-used pick for similar trip type
4
One tap to confirm → land on /trip/[id]

Data Model — Where the Role Lives

New column on trip_members

+
trip_members.role_preference TEXT NOT NULL DEFAULT 'helping_out'
Values: all_in | helping_out | just_here | vibes_only
ℹ
This is separate from the existing role column (host/member) — that one is about permissions. This one is about UX preference.
+
user_profiles.default_role_preference optional
Remembers last pick to suggest smart defaults on new trips.

What Each Role Sees on /trip/[id]

🔥 All In

✓
Default tab: Itinerary — full day-by-day editor with create/edit/reorder
✓
Sub-nav: Itinerary → Expenses → Packing → Notes → Meals → Group all 6
✓
Notifications: All on by default (group joins, new notes, event changes)
✓
Special: "Nudge the group" button surfaces on every tab

🙌 Helping Out

✓
Default tab: Itinerary — read-first view (create events possible but de-emphasized)
✓
Sub-nav: Itinerary → Packing → Expenses → Notes → Meals → Group
✓
Notifications: On for events & tasks; off for pure discussion noise
✓
Special: "Claim a task" card on trip hub showing open All-In tasks

🎟️ Just Here

✓
Default tab: Expenses — home screen is Splitwise-grade
✓
Sub-nav: Expenses → Itinerary (today-only) → Group → Packing → Notes → Meals all 6, reordered
✓
Notifications: Day-of reminders only + new expense settles
✓
Special: Events default to one-tap opt-in (no auto-yes); packing surfaced as opt-in upsell on hub

✌️ Vibes Only

✓
Default tab: Itinerary in Today-mode — single hero card for next event + amount owed
✓
Sub-nav: Itinerary → Expenses → Group → Packing → Notes → Meals all 6, reachable
✓
Notifications: Morning-of only. Nothing else.
✓
Special: A gentle "Want the full trip view?" link at bottom — opt-in escape hatch
