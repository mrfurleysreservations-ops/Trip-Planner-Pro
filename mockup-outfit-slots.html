Outfit builder — slot-based concepts

Five takes on the same idea: items auto-sort into slots (Head / Top / Bottom / Shoes / Accessories), event details collapse, and the outfit space gets bigger. Same outfit across all five — Paso Robles Day 2 daytime. Look through them and tell me which parts you like.

Concept A
Paperdoll column
Slots stacked head-to-toe. Each empty slot is a soft "tap to add". Reuse chips appear inside the tapped slot, filtered to only that slot's items — no scrolling past irrelevant stuff.

Decisions I'd need from you before building

1. Which concept (or mix)? My lean is Concept F (hybrid) — slot grid on top for at-a-glance status, reuse strip in the middle, smart input at the bottom. But tell me what feels right.

2. Dresses = Top, or a separate "One-piece" slot? You said sundress = top. Simplest. But if a dress fills the whole outfit (no bottom needed), should the Bottom tile go away? Three options:
  (a) Sundress just goes in Top, Bottom stays empty (fine if you're ok with it).
  (b) Typing "dress" unlocks a special "One-piece" mode that hides Bottom.
  (c) Rename the Top slot to "Top / One-piece" and the user mentally handles it.

3. How much event detail stays visible by default? Current mock shows the collapsed pill with: dress code · 3 event icons · date. Alternative: show just time range ("8:30 AM → 1:15 PM"). Alternative: show nothing except "▸ Events (3)". Which feels right for packing-on-a-phone?

4. Do you want the "Borrow whole outfit from another day" flow? (From the last mockup.) Still useful? Or does the new slot-based reuse kill the need?

5. Dress-code suggestions — keep, kill, or merge? Right now "Suggested for Smart Casual" is a separate chip row. In slot UI, those suggestions could show inside the slot's drawer/panel. Cleaner. OK to fold them in?
